# Detached `.deb` Signature Verification Flow

This document describes the current detached package-signature flow implemented by `src/core/repo_index.cpp`, `src/apmd/install_manager.cpp`, and `src/util/crypto/gpg_verify.cpp`.

## Source Configuration

APM reads source entries from:

- `/data/apm/sources/sources.list`
- `/data/apm/sources/sources.list.d/*.list`

Detached package signature policy is controlled per source:

```text
deb [deb-signatures=required|optional|disabled] https://example.com/debian bookworm main
```

Recognized values:

- `required`
- `optional`
- `disabled`
- aliases for disabled: `no`, `false`, `0`

Unknown values fall back to `disabled`.

## What This Is and Is Not

This policy is about detached signatures for individual downloaded package files.

It is separate from Release/InRelease verification, which is controlled by `trusted=`.

Current trust layers are:

1. Release or InRelease verification for repository metadata
2. Release checksum verification for `Packages.gz` / `Packages`
3. Package checksum verification for the downloaded `.deb`
4. Optional or required detached `.deb` signature verification

## Install-Time Flow

For each package selected during daemon-backed install:

1. Download or locate the `.deb`
2. Verify the package checksum from `Packages`
   - prefer SHA256
   - fall back to MD5 when available
3. Determine detached signature policy from the package's source
4. If policy is not `disabled`, attempt detached verification
5. Cache successful detached verification by `.deb` SHA256

## Policy Resolution

`install_manager.cpp` matches package entries back to their originating source using repository metadata already attached to each package entry:

- `repoUri`
- `repoDist`

If no matching source is found, detached signature policy falls back to `Disabled`.

## Signature Discovery

When detached verification is enabled, APM looks for a signature in this order:

1. `<debPath>.asc`
2. `<debPath>.gpg`
3. repository download of `.asc`
4. repository download of `.gpg`

`.asc` is preferred over `.gpg`.

If no detached signature is available:

- `required`: install fails
- `optional`: install continues with a warning

## Verification Engine

Verification is performed by `verifyDetachedSignature(...)` in `src/util/crypto/gpg_verify.cpp`.

Current characteristics:

- detached OpenPGP signature verification
- RSA + SHA256 support
- trusted keys loaded from `/data/apm/keys`
- matched key fingerprint can be returned for cache/audit output

## Signature Cache

Location:

- `/data/apm/pkgs/sig-cache.json`

Key:

- SHA256 of the `.deb` content

Stored metadata includes:

- signature type
- signature source
- signature path
- matched key fingerprint

A cache hit skips repeated detached verification for the same package payload bytes.

## Behavior by Policy

### `deb-signatures=required`

- detached signature must exist
- detached verification must succeed
- install aborts on failure

### `deb-signatures=optional`

- detached signature is attempted when available
- install may continue if missing or invalid
- warnings are logged

### `deb-signatures=disabled`

- detached verification is skipped
- package checksum verification still happens

## Operational Notes

- trusted keys are imported with `apm key-add <file.asc|file.gpg>`
- cache can be inspected with `apm sig-cache show`
- cache can be cleared with `apm sig-cache clear`

## Known Limitations

- RSA + SHA256 is the active detached-signature path
- no revocation checking
- no cache TTL
- detached verification depends on repositories actually publishing matching signature files
- detached verification is additional hardening, not a substitute for Release verification
