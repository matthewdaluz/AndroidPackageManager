APM Security Review Notes (Pre-Open Beta)
========================================

Scope
-----
- Manual review of project source under src/, excluding third-party OpenSSL headers.
- Focus: security vulnerabilities, privilege escalation paths, and release-blocking risks.

Findings (highest severity first)
--------------------------------

1) CRITICAL: Unsafe tar extraction allows write outside destination (Fixed)
   - Impact:
     - Any .deb or manual archive can write to arbitrary paths via absolute paths
       or ".." traversal inside tar entries. This can overwrite system files,
       install binaries, or drop scripts executed later. On a root-running daemon,
       this is full device compromise.
     - Tar extraction may preserve dangerous metadata (permissions, suid bits,
       device nodes) depending on tar implementation.
   - Evidence:
     - tar is invoked with "tar -xf" only, no path validation or safety flags.
   - References:
     - src/core/tar_extractor.cpp:37
     - src/apmd/install_manager.cpp:943
     - src/apm/apm.cpp:671
   - Suggested mitigation:
     - Use a safe tar extraction library or add strict validation for every entry.
     - Enforce: no absolute paths, no ".." components, no symlink escapes.
     - Consider tar flags to disable ownership/perms/device nodes (platform-specific).
     - Extract to a temp dir and validate resulting paths before moving into place.

2) CRITICAL: Package names used directly in filesystem paths (Fixed)
   - Impact:
     - Malicious package names can escape /data/apm using "../" or absolute paths.
     - Can overwrite arbitrary files or directories during install/remove.
   - Evidence:
     - install root computed as joinPath(baseDir, pkg.packageName)
     - .deb cache file name uses pkg.packageName directly
     - manual .deb install uses control file Package without validation
   - References:
     - src/apmd/install_manager.cpp:66
     - src/apmd/install_manager.cpp:1483
     - src/apm/apm.cpp:686
   - Suggested mitigation:
     - Enforce a strict whitelist for package names (e.g., Debian rules or a safe
       subset: [a-z0-9][a-z0-9+.-]+).
     - Reject names with '/', '\\', or ".." sequences.
     - Normalize and validate before any path construction or logging.

3) HIGH: Command injection via system() with unquoted package names
   - Impact:
     - If a package name contains shell metacharacters, system() executes attacker
       controlled commands. This is a remote code execution vector via IPC.
   - Evidence:
     - pm uninstall commands are built via string concatenation.
   - References:
     - src/apmd/apk_installer.cpp:420
     - src/apmd/apk_installer.cpp:437
     - src/apmd/factory_reset.cpp:114
   - Suggested mitigation:
     - Use exec* with argv arrays and no shell; or strictly validate package
       names with Android package name regex before constructing any command.
     - Avoid system() for anything that includes user-controlled input.

4) HIGH: Zip-slip on module ZIP extraction
   - Impact:
     - A ZIP file can write files outside the intended extraction directory using
       "../" in filenames, leading to file overwrite or code injection.
   - Evidence:
     - unzip is invoked with no validation of entry paths.
   - References:
     - src/ams/module_manager.cpp:314
   - Suggested mitigation:
     - Use a ZIP library and validate each entry path.
     - Reject absolute paths, ".." segments, and symlinks pointing outside.
     - Consider extracting to a temp dir then validating allowed outputs.

5) HIGH: .deb ar member names written without sanitization
   - Impact:
     - Crafted .deb with ar member names like "../..." can write outside the
       output directory during extraction of control/data tars.
   - Evidence:
     - Member names are used directly to build output paths.
   - References:
     - src/core/deb_extractor.cpp:160
     - src/core/deb_extractor.cpp:191
   - Suggested mitigation:
     - Validate ar member names (no '/', '\\', or ".." segments) and require
       a strict allowlist ("control.tar.*" and "data.tar.*" only).
     - Reject any unexpected members with unsafe paths.

6) MEDIUM: Session token generation uses non-cryptographic RNG
   - Impact:
     - Session tokens may be predictable on platforms where random_device is
       deterministic or weak, allowing session hijacking.
   - Evidence:
     - std::mt19937 seeded from std::random_device for token generation.
   - References:
     - src/apmd/security_manager.cpp:77
     - src/apmd/security_manager.cpp:589
   - Suggested mitigation:
     - Use a cryptographic RNG (e.g., OpenSSL RAND_bytes or OS-provided CSPRNG).
     - Avoid mt19937 for any security token.

Notes
-----
- No automated tests were found (no tests/ or *_test* files). Consider adding
  regression tests for archive extraction path traversal, package name validation,
  and IPC auth/command execution flows.
- Findings are based on current codepaths in apm/apmd/amsd. Any upstream validation
  not present here should be confirmed in review before release.
